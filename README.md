# HPCA 2027 Reviewer–Paper Matching

Automated reviewer-to-paper affinity scoring and assignment for the HPCA
2027 program committee. Given each reviewer's declared subject areas plus
their recent DBLP publication history, and each paper's title/abstract/
topics, this builds SPECTER2 embeddings for both sides and produces either
a ranked reviewer list per paper or a full load-capped assignment across
the whole paper set. See `hpca2027-matching-brief.md` for the original
design brief.

## Pipeline

```
reviewer CSV ──▶ reviewers.py ──▶ dblp.py (titles) ──▶ build_fingerprints.py ──▶ fingerprints.json
                                                                                       │
paper JSON ───▶ paper_matching.py (title+abstract+topics) ─────────────────────────────┤
                                                                                       ▼
                                                        score_papers.py  or  assign_reviewers.py
```

1. **`reviewers.py`** parses the PC acceptance-form CSV into `Reviewer` records (email, tier, declared areas, DBLP link).
2. **`dblp.py`** turns a reviewer's DBLP PID into their recent publication titles (cached, rate-limited).
3. **`build_fingerprints.py`** embeds each reviewer (titles + declared areas) with SPECTER2 into a 768-dim vector, cached in `fingerprints.json`.
4. **`paper_matching.py`** does the same for papers (title+abstract, topics) — a shared module, not a standalone script.
5. **`score_papers.py`** / **`assign_reviewers.py`** consume both fingerprint caches to rank or assign reviewers per paper.
6. **`main.py`** and **`nearest_neighbors.py`** are diagnostic/exploration tools used while building the above.

## Setup

The working Python environment is a venv at `~/envs/hpca-matching` (not the
system Python) with `torch` + CUDA, `transformers`, `adapters` (AdapterHub —
needed to load the SPECTER2 proximity adapter), and `numpy`. See
`requirements.txt` for the package list; the CUDA-specific torch build isn't
reproducible via a plain `pip install -r requirements.txt`, so that file is
documentation, not a working install script. Run everything as:

```bash
~/envs/hpca-matching/bin/python3 <script>.py [args]
```

Scripts that don't touch SPECTER2 (`main.py`, `reviewers.py` as a library)
work fine under plain `python3` too, since they only need `requests`.

## Data files

**Inputs** (not generated by any script):
- `HPCA'27 PC Member Acceptance Form (Responses) - Form Responses 1.csv` — Google Forms export of PC-member responses. Contains real names/emails — see the note on sensitive data below.
- `hpca2027-data.json` — HotCRP paper export (title, abstract, topics, authors, `pc_conflicts`).
- `dblp_pubs_cache.json` — a colleague's pre-built, read-only DBLP publication cache (richer/uncapped; checked before hitting the network).

**Generated caches** (safe to delete; scripts rebuild them incrementally):
- `dblp_cache.json` — our own writable DBLP title cache, keyed by DBLP PID.
- `fingerprints.json` — reviewer SPECTER2 fingerprints, keyed by email.
- `paper_fingerprints.json` — paper SPECTER2 fingerprints, keyed by pid.

## Scripts

### `main.py` — DBLP title fetcher (diagnostic)
Prints each reviewer's recent DBLP titles. Useful for validating the DBLP
fetch/cache path in isolation.
```bash
python3 main.py --limit 5           # validate on a handful first
python3 main.py --years 2           # narrower year window
```

### `build_fingerprints.py` — build reviewer fingerprints
For each accepted reviewer, pools SPECTER2 embeddings of their DBLP titles
(within `--years`, no count cap by default) and one document summarizing
their declared areas + keywords, weighted by `--area-weight`. Reviewers with
no DBLP PID or no titles in the window fall back to an area-only
fingerprint. Incremental — a reviewer already in `fingerprints.json` isn't
recomputed.
```bash
~/envs/hpca-matching/bin/python3 build_fingerprints.py --limit 10   # validate first
~/envs/hpca-matching/bin/python3 build_fingerprints.py              # full run (~seconds on a 4090)
```
Key flags: `--years` (default 4), `--max-titles` (default: no cap), `--area-weight` (default 1.0).

### `nearest_neighbors.py` — reviewer/reviewer similarity (diagnostic)
Prints each reviewer's top-N most similar other reviewers by fingerprint
cosine similarity — a sanity check that fingerprints cluster sensibly by
topic, and a quick way to eyeball one reviewer's profile.
```bash
~/envs/hpca-matching/bin/python3 nearest_neighbors.py --email someone@example.com
```

### `score_papers.py` — rank reviewers per paper (unconstrained)
Fingerprints each paper (title+abstract, topics), then for each paper prints
the top-N reviewers by cosine similarity, independently per paper — no
awareness of other papers or per-reviewer load. Two filters apply before
ranking: **COI** (exclude any reviewer in the paper's `pc_conflicts`) and the
**area gate** (reviewer's primary/secondary must overlap the paper's topics,
case-insensitive; `--no-area-gate` disables it).
```bash
~/envs/hpca-matching/bin/python3 score_papers.py --pid 8 --top 10
```
Use this for quick "who best matches this paper" lookups; use
`assign_reviewers.py` for an actual assignment that respects reviewer load.

### `assign_reviewers.py` — global load-capped assignment
Solves one assignment across **all** papers at once, respecting COI, the
area gate, and a per-reviewer cap on total papers (`--light-cap` / `--full-cap`
by tier) and a target reviewer count per paper (`--reviewers-per-paper`).
Uses paper-proposing deferred acceptance (the "Hospital/Residents" stable
matching algorithm — the same kind of algorithm behind the US medical
residency match): each paper proposes to its best eligible reviewers first;
a reviewer holds the best offers received so far, up to their cap, bumping a
worse one whenever a better offer arrives. Guaranteed to produce zero
"blocking pairs" (no reviewer/paper pair that would both rather swap into
each other over a current match) — self-checked and reported every run.
```bash
~/envs/hpca-matching/bin/python3 assign_reviewers.py --light-cap 1 --full-cap 2 --reviewers-per-paper 6
```
If demand exceeds what the eligible reviewer pool can supply, papers come up
under-filled (flagged inline) and a **shortage report** prints at the end,
grouped by subject area — pool size (reviewer count, tier split, total
capacity) plus which under-filled papers need that area, sorted worst-first.
A multi-topic paper's shortfall is attributed to every one of its topic
areas (not split), since there's no clean way to know which one is the true
bottleneck — the report tells you where to look, not an exact partition.

## Known data-quality notes

- **Duplicate submissions**: ~35 emails have more than one form submission (people who resubmitted, sometimes changing their tier or accept/decline status). `reviewers.load_reviewers` collapses to the latest-timestamp row per email (case-insensitive) *before* filtering declines, so a later decline correctly overrides an earlier accept and vice versa.
- **Area-string casing**: HotCRP's paper topics don't always match the CSV's area spelling exactly (e.g. `"Memory Systems"` vs `"Memory systems"`). All area-matching code (the area gate, the shortage report) normalizes case.
- **~13% of accepted reviewers have no DBLP link**: they get an area-only fingerprint (no title signal).

## Sensitive data

The acceptance-form CSV contains real PC members' names and email
addresses; `fingerprints.json` and the DBLP caches key off those same
emails. Treat these as sensitive — don't publish them somewhere public
without deliberately deciding to.
